# CLINC150 demo — abstention / route-to-human

[CLINC150](https://github.com/clinc/oos-eval) is 150 fine-grained user intents
across 10 domains, plus an explicit **out-of-scope (OOS)** set of queries that
belong to none of them. That makes it an ideal showcase for this package: the
model learns the 150 in-scope intents, and the OOS queries are exactly what a
calibrated, abstaining classifier should decline and route to a human.

This demo runs **fully offline** with the torch-free TF-IDF encoder (no model
download). Swap `--encoder-kind tfidf` for the default sentence-transformers
encoder for higher quality if you have network access.

## Run it

```bash
# 1. Build the CSVs (downloads CLINC150 once, ~2.5 MB).
#    Drop --max-classes/--per-class for the full 150-intent dataset.
python examples/clinc150/prepare.py --out examples/clinc150/build \
    --max-classes 40 --per-class 60

B=examples/clinc150/build

# 2. Train offline. target-precision is the operating point: the minimum
#    accuracy required on accepted items (the threshold is tuned for it).
text-classifier-train --items $B/items_train.csv --classes $B/classes.csv \
    --out $B/model --encoder-kind tfidf --target-precision 0.99

# 3. Evaluate on in-scope test — coverage, accuracy, calibration (Brier/ECE),
#    a risk-coverage curve, and a per-class breakdown, written to JSON.
text-classifier-eval --model $B/model --input $B/items_test.csv \
    --output $B/in_scope_report.json

# 4. Show abstention on out-of-scope queries: the desired behavior is to abstain.
text-classifier-infer --model $B/model --input $B/items_oos.csv \
    --output $B/oos_preds.csv
python examples/clinc150/check_oos.py $B/oos_preds.csv
```

## What it demonstrates

The abstention threshold is the knob. It is tuned to hold a target **accuracy on
accepted** in-scope items; raising it makes the system more conservative, which
also rejects more out-of-scope queries — at the cost of in-scope coverage. This
is the core risk–coverage trade-off a human-in-the-loop system lives on.

Illustrative results from the 40-intent, TF-IDF subsample above (your exact
numbers will vary with the subsample, encoder, and seed):

| `--target-precision` | in-scope coverage | in-scope accuracy | OOS rejected |
|----------------------|-------------------|-------------------|--------------|
| 0.90                 | 100%              | 95.8%             | 1.0%         |
| 0.99                 | 76.5%             | 96.5%             | 88.6%        |
| 0.999                | 58.8%             | 97.9%             | 96.7%        |

Takeaways for reading the demo:

- **The operating point is a real choice.** At 0.90 the threshold sits low
  (in-scope is easy), so OOS leaks through; at 0.99 the system rejects ~89% of
  OOS while still auto-handling ~77% of in-scope traffic. Pick the point from
  *your* cost of a wrong answer vs. a human review.
- **Calibration is what makes the threshold meaningful.** `in_scope_report.json`
  reports Brier/ECE and a reliability table; the threshold is only as trustworthy
  as the calibration behind it.
- **A real bi-encoder separates OOS better** at a given coverage than TF-IDF.
  This offline run is the floor, not the ceiling.

## Notes

- Class **descriptions** are humanized from the intent key
  (`bill_balance` → "A user request about bill balance."). They feed the
  description-similarity signal and the encoder fine-tuning; hand-writing richer
  descriptions is the first thing to improve for a production-grade demo.
- CLINC150 is (by construction) class-balanced in-scope, so it showcases
  abstention rather than class imbalance. For an imbalanced, many-described-class
  showcase, an arXiv abstract → primary-category demo is a better fit.
- `build/` and the downloaded `data_full.json` are git-ignored — they are
  regenerated by `prepare.py`.
